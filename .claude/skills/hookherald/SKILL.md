# HookHerald

HookHerald bridges external systems to Claude Code sessions. It works two ways:

1. **Watchers** — scripts that run on an interval; their stdout becomes a notification in Claude Code
2. **Webhooks** — external systems POST to a local router, which forwards to the right Claude Code session

```
Watchers:  Script stdout --> Channel --> Router --> Channel --> Claude Code
Webhooks:  HTTP POST -----> Router ----------------> Channel --> Claude Code
```

The **router** is a central HTTP server (default port 9000). The **channel** is an MCP server that runs inside Claude Code, spawned automatically via `.mcp.json`. Watchers are configured in `.hookherald.json` and managed by the channel.

## Before You Start

Before doing anything with HookHerald, check that the prerequisites are in place. Run these checks and fix anything that's missing — don't skip ahead.

### Is HookHerald installed?

```bash
hh --help
```

If this fails with "command not found", install it:

```bash
npm install -g hookherald
```

Requires Node.js >= 18.

### Is the router running?

```bash
hh status
```

If this connects and shows a table (even an empty one), the router is up. If it fails with "connection refused" or similar, start it:

```bash
hh router --bg
```

The router runs in the background and writes its PID to `~/.hookherald/router.pid`. Stop it later with `hh router stop`.

### Is the project initialized?

Check if `.hookherald.json` exists in the project directory. If not, initialize:

```bash
cd ~/my-project
hh init
```

This creates two files:
- **`.mcp.json`** — tells Claude Code how to spawn the channel (MCP server). Auto-generated, don't edit by hand.
- **`.hookherald.json`** — your project config. This is the file you edit.

The slug (project identifier used for routing) is auto-detected from your git remote. Override with `hh init --slug mygroup/myproject`.

After init, verify the router can see the project:

```bash
hh status
```

### Start Claude Code

Launch Claude Code from the same directory where `.mcp.json` lives:

```bash
claude
```

The channel starts, registers with the router, and appears on the dashboard at `http://127.0.0.1:9000`.

### Add a watcher

Edit `.hookherald.json` — no restart needed, the channel hot-reloads automatically:

```json
{
  "slug": "mygroup/myapp",
  "router_url": "http://127.0.0.1:9000",
  "watchers": [
    { "command": "./check-ci.sh", "interval": 15 }
  ]
}
```

The watcher runs immediately, then every 15 seconds.

## Configuration

### .hookherald.json

This is the file you edit to configure watchers:

```json
{
  "slug": "owner/project",
  "router_url": "http://127.0.0.1:9000",
  "watchers": [
    { "command": "./my-script.sh", "interval": 30 },
    { "command": "kubectl get pods -o json", "interval": 60 }
  ]
}
```

- **slug**: Project identifier for routing. Format: `owner/project`.
- **router_url**: Where the router is running.
- **watchers**: Array of `{ command, interval }`. Command is any shell command. Interval is in seconds.

### Hot reload

When you save `.hookherald.json`, the channel detects the change within 200ms. Removed watchers stop, new watchers start immediately. No need to restart Claude Code.

### .mcp.json

Auto-generated by `hh init`. Contains the MCP server command and environment variables. Don't edit this file — if you need to change the slug or router URL, edit `.hookherald.json` instead and re-run `hh init`.

## Writing Watcher Scripts

Watchers are the core extension point. Any script or command that writes to stdout can be a watcher.

### The contract

- **stdout = send**: Non-empty stdout gets wrapped in an envelope and delivered to Claude Code as a notification
- **empty stdout = skip**: Nothing happens, no notification sent
- **exit code doesn't matter**: HookHerald captures stdout regardless of exit code
- **JSON is auto-parsed**: If stdout is valid JSON, it stays structured in the notification. Otherwise it's sent as a string.
- **The script owns dedup and state**: HookHerald doesn't diff or deduplicate — the script decides when to fire

Scripts run via `sh -c` with a 60-second timeout, from the project root (where `.hookherald.json` lives).

### Minimal template

```bash
#!/bin/bash
RESULT=$(your-command-here)
if [ -z "$RESULT" ]; then exit 0; fi
echo "$RESULT"
```

### With state tracking (dedup)

Most watchers should only notify when something changes. Use a state file in `/tmp/`:

```bash
#!/bin/bash
STATE_FILE="/tmp/hh-my-watcher-state"

CURRENT=$(your-command-here)
LAST=$(cat "$STATE_FILE" 2>/dev/null)

if [ "$CURRENT" = "$LAST" ]; then exit 0; fi

echo "$CURRENT" > "$STATE_FILE"
echo "$CURRENT"
```

To force a re-report, delete the state file: `rm /tmp/hh-my-watcher-state`

### JSON output

Claude works better with structured data. Use `jq` to format output:

```bash
echo "$CURRENT" | jq '{
  status: .status,
  summary: { total: (.items | length) }
}'
```

### Testing your script

Always test standalone before plugging into `.hookherald.json`:

```bash
./my-watcher.sh        # Should produce output (first run, no state)
./my-watcher.sh        # Should produce nothing (state unchanged)
rm /tmp/hh-my-watcher-state
./my-watcher.sh        # Should produce output again
```

If the standalone output looks right, add it to `.hookherald.json`. The channel picks it up automatically via hot reload.

### What Claude sees

When a watcher fires, Claude receives a channel notification:

```
<channel source="webhook-channel">
{
  "project_slug": "mygroup/myapp",
  "source": "./my-watcher.sh",
  "output": { ... your script's stdout ... }
}
</channel>
```

For full example scripts (GitHub Actions, Kubernetes, GitLab CI, deployment rollouts), read `references/watcher-examples.md`.

## Webhooks

For systems that can POST to you (GitHub, GitLab, Jenkins), use webhooks instead of watchers.

### Sending a webhook

POST to the router with a `project_slug` field matching a registered session:

```bash
curl -X POST http://127.0.0.1:9000/ \
  -H "Content-Type: application/json" \
  -d '{"project_slug": "mygroup/myapp", "event": "push", "branch": "main"}'
```

### Authentication (opt-in)

By default, no auth is required (localhost is trusted). To require a token:

```bash
hh router --secret my-secret-token
```

Then include the token in webhook requests via `X-Webhook-Token` or `X-Gitlab-Token` header. Configure the same secret in your CI system's webhook settings.

### When to use webhooks vs watchers

| Scenario | Use |
|----------|-----|
| Source can POST to you | Webhook |
| You need to poll/check something | Watcher |
| Instant notification needed | Webhook |
| No network access to router | Watcher |

## Deployment

### Router options

```bash
hh router                    # Foreground
hh router --bg               # Background (PID in ~/.hookherald/router.pid)
hh router stop               # Stop background router
hh router --port 8080        # Custom port
```

### Docker

```bash
docker run -d --network host shoofio/hookherald
```

### Dashboard

Live management UI at `http://127.0.0.1:9000` — shows sessions, events, watcher tags, latency. SSE-powered, updates in real time.
